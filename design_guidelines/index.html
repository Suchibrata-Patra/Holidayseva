Airframe UI

A compact, zero-dependency CSS design system built for modern booking and marketplace interfaces. Drop in one file — tokens flow directly into components.

Web Responsive 15+ Components 0 Dependencies

Design Principles #

Airframe is built on three core ideas: token-first theming, composable primitives, and zero runtime overhead. Every visual decision — color, space, radius, shadow — is expressed as a named CSS custom property. Components are thin wrappers that consume those tokens, meaning a single variable change propagates consistently across the entire system.

The system intentionally avoids JavaScript-heavy patterns. Dropdowns, modals, and interactive states rely on minimal vanilla JS with no external dependencies. This keeps bundle size near zero while ensuring the system works in any environment — static sites, server-rendered apps, and SPAs alike.

ℹ
Token-first rule

Always reference a design token like var(--color-primary), never a raw value like #FF385C. This ensures every component automatically participates in theming and future palette updates require editing exactly one location.

Architecture

The system is structured in two layers. Foundation tokens define the raw palette, type scale, and spacing rhythm. Semantic tokens map those primitives to intent — --color-primary may resolve to a red today and can be repointed to a blue without touching components.

Components are authored with a BEM-like flat class structure to keep specificity low and overrides predictable. Modifier classes use double-dash suffixes (btn--sm, avatar--lg) to signal variation, while state classes use data attributes where possible for easier JS targeting.

Color #

All colors are defined as CSS custom properties on :root. The palette is split into three layers: brand colors expressing product identity, semantic colors communicating meaning at a glance, and neutral ink tones providing accessible text and surface hierarchies.

Light and dark mode are handled by overriding the same token names inside a @media (prefers-color-scheme: dark) block — no separate class toggling required. Components never reference hardcoded values; they consume tokens exclusively.

Brand

Brand colors should be used for primary interactive elements, CTAs, and brand expressions. Avoid using --color-primary for purely decorative purposes where it would compete with actionable elements.

⚠
Contrast requirements

All text-on-color combinations must meet WCAG AA (4.5:1 for normal text, 3:1 for large text). Run contrast checks before introducing new color pairings. The semantic tokens above have been pre-validated for white backgrounds at AA level.

Typography #

The type scale uses clamp() for fluid sizing — values interpolate smoothly between a minimum and maximum as the viewport grows, eliminating the need for breakpoint-driven overrides. All steps are expressed as CSS custom properties so they compose predictably across components.

Body copy defaults to DM Sans for its geometric, legible forms at small sizes. Numeric data and code snippets use DM Mono for visual rhythm and disambiguation. Headlines may optionally use a serif display face to signal editorial hierarchy.

Spacing #

Spacing follows a base-4 geometric scale. All layout gaps, padding values, and margins should be drawn exclusively from these tokens to maintain visual rhythm. Never use arbitrary pixel values — if an existing step doesn't fit, the layout needs reconsideration, not a one-off value.

The scale is intentionally dense at small values (4–16px) to support compact UI chrome like toolbars and table rows, and progressively generous at larger values (48–80px) for section separation and page margins.

Shadows & Elevation #

Shadows communicate elevation and focus. They should reinforce perceived z-axis position, not add decoration. Use the lowest elevation shadow that establishes sufficient separation from the background — heavier shadows are reserved for dialogs, popovers, and focused elements that need strong spatial separation.

Border Radius #

Border radius tokens define the system's softness vocabulary. Smaller radii (--r-xs, --r-sm) are reserved for dense inline elements like tags and code spans. Larger radii (--r-lg, --r-xl) are used on cards and containers. --r-pill is only for tags, chips, and pills — never on rectangular content blocks.

Button #

Buttons communicate available actions to the user. Each page should have at most one primary button visible at a time to preserve clear hierarchy. Secondary and ghost variants exist for less prominent actions. Destructive variants are reserved for irreversible operations and should be used sparingly.

All button sizes share the same font family and weight. Padding scales proportionally — do not resize a button by overriding padding directly; use the provided size modifier classes instead.

Badge #

Badges provide compact labeling for status, counts, and categorical information. They are always non-interactive — if a badge needs to be tappable, it should be refactored into a tag or chip. Limit badge use to 1–2 per list item to avoid visual noise.

Avatar #

Avatars represent users with a photo or initials fallback. Always provide an alt attribute on the image for accessibility. Initials should be two characters maximum — truncate from the first letter of each name segment.

The avatar-group wrapper applies overlapping margins automatically. Groups larger than 5 should show the first 4 and a +N overflow indicator to avoid visual clutter.

Gallery #

The Gallery component renders a 5-photo hero grid. The hero image spans full height on the left with four thumbnails filling a 2×2 grid on the right. On mobile, only the hero image is shown. Always provide a "Show all photos" affordance for listings with more than 5 images.

Booking Card #

The Booking Card is a compact reservation widget used in listing sidebars. Check-in and checkout fields share a connected border to visually group related inputs. The guest picker expands inline via the Dropdown component. Always show price-per-night prominently before the reserve CTA.

Calendar #

The Calendar renders two months side by side for date range selection. Click a start date then an end date — the range highlights automatically. Past dates are visually disabled and non-interactive. Month navigation arrows are present on each panel.

Dropdown #

Dropdowns expose a list of options below a trigger. They close automatically on outside click or Escape key. A single page should not contain more than two open dropdowns simultaneously; if many filters are needed, consider a filter panel instead.

Review Card #

Review Cards display guest feedback in a consistent, scannable format. Body text is truncated at three lines with an expand affordance. Always show the reviewer's avatar, name, date, and star rating. Avoid truncating text in accessibility-critical contexts — show the full content or provide a clear "Read more" link.

Modal #

Modals interrupt the user's current task to require acknowledgment or a decision. Use them only for high-stakes confirmations — do not use modals for information that can be shown inline. Modals always include a close button, respond to Escape, and close on backdrop click. Body scroll is locked while a modal is open.

💡
Accessibility

Trap focus within the modal when open. Use role="dialog" and aria-modal="true". Restore focus to the triggering element on close.

Navigation Buttons #

Navigation buttons provide directional traversal within carousels, image galleries, and paginated content. They are circular icon-only buttons and must always be paired (previous and next). Disable the previous button at the first item and the next button at the last item rather than hiding them, to avoid layout shift.

Price Card #

The Price Card is a compact sticky footer alternative to the full Booking Card for mobile viewports. It shows the nightly rate prominently alongside a single reserve CTA. On scroll-down it appears fixed to the bottom of the screen; on scroll-up it reveals a more detailed pricing breakdown.
